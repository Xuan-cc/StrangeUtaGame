# 领域层设计

## 概述

领域层定义**核心业务概念**和**业务规则**，是整个应用的核心。它不依赖任何外部框架或库，纯 Python 实现，确保可测试性和可移植性。

## 领域模型

### 实体（Entity）

实体具有**唯一标识**和**生命周期**，可以被创建、修改、持久化。

#### LyricLine（歌词行）

**职责**：表示一行歌词及其时间标签、注音信息、节奏点配置

**属性**：
- `id`: 唯一标识符（UUID）
- `text`: 原始文本
- `chars`: 拆分的字符列表
- `timetags`: 时间标签列表（按时间排序）
- `rubies`: 注音列表
- `checkpoints`: 节奏点配置列表（与 chars 一一对应）
- `is_line_end`: 行尾标记（表示此行末尾是否有休止/长音）

**业务规则**：
1. 时间标签必须按时间递增
2. 字符与时间标签可一对多（连打）
3. 注音可覆盖连续多个字符
4. 每个字符对应一个节奏点配置（CheckpointConfig）
5. 节奏点数量（check_count）表示该字符需要击打几次
6. 句尾字符（行末）通常有更长的时间或特殊处理

**行为**：
- 添加时间标签（自动排序）
- 查询指定字符的时间标签
- 查询指定字符的注音
- 计算行时长
- 获取指定字符的节奏点配置
- 批量设置节奏点数量（自动检查功能）

#### Singer（演唱者/角色）

**职责**：表示一个演唱者或角色（用于多声部和声场景）

**属性**：
- `id`: 唯一标识符
- `name`: 演唱者名称（如 "初音ミク"、"合唱"、"和声"）
- `color`: 显示颜色（用于区分不同演唱者）
- `is_default`: 是否为默认演唱者
- `display_priority`: 显示优先级（数字越小越优先显示）

**业务规则**：
1. 每个项目至少有一个演唱者（默认演唱者1）
2. 演唱者可以有独立的歌词轨道
3. 同一时间点可以有多个演唱者同时演唱
4. 演唱者可以临时禁用（静音/隐藏）

**行为**：
- 重命名
- 修改颜色
- 启用/禁用

#### Project（项目）

**职责**：作为聚合根，管理所有歌词行和演唱者的一致性

**属性**：
- `id`: 唯一标识符
- `audio_path`: 关联的音频文件路径
- `lines`: 歌词行列表（所有演唱者的歌词混合存储，通过 singer_id 关联）
- `singers`: 演唱者列表
- `metadata`: 元数据（标题、艺术家等）

**业务规则**：
1. 歌词行按时间顺序排列（可选，允许乱序以支持和声）
2. 每行有唯一 ID，并关联到特定演唱者
3. 项目必须关联有效音频文件
4. 必须至少有一个演唱者（默认）

**行为**：
- 添加演唱者
- 删除演唱者（级联删除其歌词或转移到其他演唱者）
- 添加歌词行（指定演唱者）
- 根据演唱者获取歌词行
- 根据时间获取所有演唱者的当前行（支持和声场景）
- 验证项目数据有效性

### 值对象（Value Object）

值对象**无唯一标识**，由属性值决定相等性，创建后不可变。

#### Checkpoint（节奏点/检查点）

**职责**：表示一个需要击打的时间点，对应用户一次按键操作

**属性**：
- `timestamp_ms`: 时间戳（毫秒）
- `char_idx`: 对应字符索引（在 LyricLine.chars 中的位置）
- `checkpoint_idx`: 在该字符内的第几个节奏点（从 0 开始）

**相等性**：timestamp_ms、char_idx、checkpoint_idx 都相同

**约束**：
- 时间非负
- char_idx 在有效范围内
- 同一字符内的节奏点按时间递增

**与 TimeTag 的区别**：
- TimeTag 是时间标签的存储形式
- Checkpoint 是打轴时的操作单位
- 一个字符可以有多个 Checkpoint（连打场景）

#### CheckpointConfig（节奏点配置）

**职责**：配置每个字符的节奏点数量和类型

**属性**：
- `char_idx`: 字符索引
- `check_count`: 节奏点数量（需要击打几次，默认 1）
- `is_line_end`: 是否是句尾字符（行末，默认 False）
- `is_rest`: 是否是休止符（默认 False）

**业务含义**：
- `check_count`: 该字符对应的读音需要击打几次
  - 例：「赤」(あか) → check_count = 2（あ・か）
  - 例：「い」→ check_count = 1
  - 例：「ー」→ check_count = 0 或 1（可配置）
- `is_line_end`: 标记行末字符，通常有长音或休止
- `is_rest`: 特殊标记，表示这是一个休止符而非歌词字符

**相等性**：所有属性都相同

#### TimeTag（时间标签）

**职责**：表示一个时间点，**允许时间倒退**（用户修改时可能出现）

**属性**：
- `timestamp_ms`: 绝对时间（毫秒）
- `tag_type`: 标签类型（字符开始/中间/行尾）
- `singer_id`: 所属演唱者 ID（用于多声部）

**相等性**：timestamp_ms、tag_type 和 singer_id 都相同

**约束**：
- 时间非负
- 精度为毫秒
- **不强制递增**：允许时间倒退，以支持用户修改场景

**时间倒退场景**：
- 用户修改前面的歌词时间，导致比后面的晚
- 和声场景：不同演唱者的时间标签可能交错
- 系统不阻止添加，而是标记为"异常"供用户后续调整

#### Ruby（注音）

**职责**：表示汉字的注音（振り仮名）

**属性**：
- `text`: 注音文本（如 "あか"）
- `start_idx`: 起始字符索引
- `end_idx`: 结束字符索引（不包含）

**相等性**：所有属性都相同

**约束**：
- start_idx < end_idx
- 索引在有效范围内

## 业务规则详解

### 时间标签规则

**递增性（宽松规则）**：
- **不强制递增**：允许时间倒退，系统仅标记为"时间异常"供用户注意
- 同一行内的时间标签**建议**按时间递增，但不强制
- 行间时间标签可以重叠（多段ワイプ场景）
- 多演唱者场景：不同演唱者的时间标签完全独立，可任意交错

**对应关系**：
- 基础情况：一个字符对应一个时间标签
- 连打情况：一个字符对应多个时间标签（连续击打）
- 行尾休止：独立的时间标签表示行间停顿
- 多演唱者：每个演唱者有独立的时间标签集合

**精度**：
- 存储精度：毫秒（int）
- 显示精度：10 毫秒（0.01 秒）

**时间异常处理**：
- **检测**：系统检测时间倒退情况（后一个时间早于前一个）
- **标记**：在 UI 中高亮显示异常段落（如红色边框、警告图标）
- **渲染**：异常段落播放完成后高亮提示用户调整
- **不阻止**：不阻止用户继续打轴，允许先完成再调整

### 字符拆分规则

**日文文本**：
- 汉字：单独拆分（如「赤」）
- 假名：单独拆分（如「い」）
- 长音：可配置是否独立拆分（「ー」）
- 促音：可配置是否独立拆分（「っ」）

**英文文本**：
- 字母单独拆分
- 单词边界保留

**符号处理**：
- 标点符号单独拆分
- 连续空白合并为单个空格

### 注音规则

**覆盖规则**：
- 一个字符只能有一个注音
- 注音可以覆盖 1-N 个连续字符
- 支持多对一（如「昨日」→「きのう」）

**优先级**：
1. 用户手动指定的注音
2. 自动分析结果
3. 无注音（空）

### 节奏点（Checkpoint）规则

**数量计算（自动检查）**：

根据注音计算每个字符的节奏点数量：
- 汉字：注音的假名数量 = 节奏点数量
  - 例：「赤」(あか) → 2 个节奏点
  - 例：「花」(はな) → 2 个节奏点
- 平假名/片假名：通常 1 个
- 促音「っ/ッ」：可配置 0 或 1 个
- 长音「ー」：可配置 0 或 1 个
- 英文：可配置 1 个或多个

**句尾标记（is_line_end）**：
- 每行最后一个字符自动标记为句尾
- 句尾字符在导出时可能有特殊处理
- 支持手动调整句尾位置

**节奏点与时间标签的对应**：
- 基础情况：1 个节奏点 = 1 个时间标签
- 连打情况：1 个字符的 N 个节奏点 = N 个时间标签
- 未打轴：0 个时间标签（等待用户打轴）

**行尾休止**：
- 行末可配置休止符（独立的节奏点）
- 休止符在歌词显示中通常不显示
- 休止符用于控制行间停顿时间

### 字符与时间标签映射

**一对一**：
```
字符：  赤    い    花
节奏点：1     1     1
时间：  [10]  [15]  [20]
```

**一对多（连打）**：
```
字符：  赤         い    花
节奏点：2          1     1
时间：  [10][12]  [15]  [20]
        ↑   ↑
        あ   か
```

**多对一（连音）**：
```
字符：  昨    日
节奏点：2（きのう）
注音：  きの+ う
时间：  [10]  [15]
        ↑
        きのう 对应两个字
```

### 多演唱者（和声）规则

**演唱者划分**：
- 默认只有一个演唱者（演唱者1）
- 用户可添加多个演唱者（演唱者2、3...或命名如"合唱"、"和声"）
- 每个演唱者有独立的：
  - 歌词行集合
  - 时间标签集合
  - 显示颜色

**同时演唱场景**：
```
时间点：[10秒]
演唱者1：赤（高亮红色）
演唱者2：青（高亮蓝色） ← 同时高亮

时间点：[15秒]
演唱者1：い（高亮红色）
演唱者2：（无歌词）
```

**时间重叠处理**：
- 不同演唱者的时间标签完全独立
- 允许同一时间点多个演唱者同时有高亮字符
- UI 使用不同颜色区分不同演唱者
- 时间倒退仅在同演唱者内检测，跨演唱者不检测

**演唱者切换**：
- 用户可选择当前为哪个演唱者打轴
- 支持"全部显示"模式（查看所有演唱者）
- 支持"单演唱者"模式（专注一个轨道）

## 聚合边界

### Project 聚合

**聚合根**：Project
**包含实体**：LyricLine, Singer
**包含值对象**：TimeTag, Ruby, Checkpoint, CheckpointConfig

**一致性规则**：
- 所有修改必须通过 Project 进行
- 删除 Project 时级联删除所有行和演唱者
- 删除演唱者时：级联删除其所有歌词行，或转移给其他演唱者
- 行的顺序由 Project 维护（可按演唱者分组）
- 必须至少保留一个演唱者

## 验证规则

### 项目级验证

1. **音频文件存在性**：audio_path 指向的文件必须存在
2. **歌词行非空**：至少有一行歌词
3. **时间标签完整性**：每行至少有一个时间标签（除空行外）

### 行级验证

1. **时间递增性**：时间标签时间戳递增
2. **数量合理性**：时间标签数量不超过字符数量过多
3. **注音覆盖性**：注音索引不越界

### 时间标签验证

1. **非负性**：时间戳 >= 0
2. **合理性**：时间戳不超过音频时长

## 与上层交互

### Application 层如何使用

1. 通过 Project 方法创建/修改数据
2. 直接读取领域对象属性
3. 不直接修改领域对象（通过方法调用）

### 禁止的操作

- ❌ 直接修改 LyricLine.timetags 列表
- ❌ 直接修改 TimeTag.timestamp_ms
- ❌ 手动创建未关联到 Project 的 LyricLine

## 测试策略

### 单元测试范围

- 实体行为方法
- 值对象相等性
- 业务规则验证

### 测试数据

- 使用内存中的对象，不涉及文件/数据库
- 构造各种边界情况（空行、无时间标签等）
