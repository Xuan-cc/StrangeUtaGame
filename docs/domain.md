# 领域层设计

## 概述

领域层定义**核心业务概念**和**业务规则**，是整个应用的核心。它不依赖任何外部框架或库，纯 Python 实现，确保可测试性和可移植性。

## 领域模型

### 实体（Entity）

实体具有**唯一标识**和**生命周期**，可以被创建、修改、持久化。

#### LyricLine（歌词行）

**职责**：表示一行歌词及其时间标签、注音信息、节奏点配置

**属性**：
- `id`: 唯一标识符（UUID）
- `singer_id`: 所属演唱者 ID（必填，关联到 Singer）
- `text`: 原始文本
- `chars`: 拆分的字符列表（由 text 动态生成，用于索引）
- `timetags`: 时间标签列表（按时间排序）
- `rubies`: 注音列表
- `checkpoints`: 节奏点配置列表（与 chars 一一对应）
- `is_line_end`: 行尾标记（表示此行末尾是否有休止/长音）

**关于 text 和 chars 的关系**：
- `chars` 是 `text` 的拆分结果，运行时动态生成
- 持久化时两者都存储，用于数据校验（确保 chars 与 text 一致）
- 加载时优先使用存储的 chars，如果与 text 不符则重新拆分
- 拆分规则由 TextSplitter 服务处理（日文/英文/符号分别处理）

**业务规则**：
1. 时间标签必须按时间递增
2. 字符与时间标签可一对多（连打）
3. 注音可覆盖连续多个字符
4. 每个字符对应一个节奏点配置（CheckpointConfig）
5. 节奏点数量（check_count）表示该字符需要击打几次
6. 句尾字符（行末）通常有更长的时间或特殊处理

**行为**：
- 添加时间标签（自动排序）
- 查询指定字符的时间标签
- 查询指定字符的注音
- 计算行时长
- 获取指定字符的节奏点配置
- 批量设置节奏点数量（自动检查功能）

#### Singer（演唱者/角色）

**职责**：表示一个演唱者或角色（用于多声部和声场景）

**属性**：
- `id`: 唯一标识符
- `name`: 演唱者名称（如 "初音ミク"、"合唱"、"和声"）
- `color`: 显示颜色（用于区分不同演唱者）
- `is_default`: 是否为默认演唱者
- `display_priority`: 显示优先级（数字越小越优先显示）

**业务规则**：
1. 每个项目至少有一个演唱者（默认演唱者1）
2. 演唱者可以有独立的歌词轨道
3. 同一时间点可以有多个演唱者同时演唱
4. 演唱者可以临时禁用（静音/隐藏）

**行为**：
- 重命名
- 修改颜色
- 启用/禁用

#### Project（项目）

**职责**：作为聚合根，管理所有歌词行和演唱者的一致性

**属性**：
- `id`: 唯一标识符
- `lines`: 歌词行列表（所有演唱者的歌词混合存储，通过 singer_id 关联）
- `singers`: 演唱者列表
- `metadata`: 元数据（标题、艺术家、语言等）
- `audio_duration_ms`: 音频时长（毫秒，用于验证用户选择的音频是否匹配）

**业务规则**：
1. 歌词行按时间顺序排列（可选，允许乱序以支持和声）
2. 每行有唯一 ID，并关联到特定演唱者（singer_id 必填）
3. 必须至少有一个演唱者（default = true）
4. 所有时间标签必须携带 singer_id，确保多演唱者场景数据完整
5. **不存储音频路径** - 音频由用户每次使用时单独选择

**行为**：
- 添加演唱者
- 删除演唱者（级联删除其歌词或转移到其他演唱者）
- 添加歌词行（指定演唱者）
- 根据演唱者获取歌词行
- 根据时间获取所有演唱者的当前行（支持和声场景）
- 验证项目数据有效性

### 值对象（Value Object）

值对象**无唯一标识**，由属性值决定相等性，创建后不可变。

#### Checkpoint（节奏点/检查点）

**职责**：表示一个需要击打的时间点，对应用户一次按键操作

**属性**：
- `timestamp_ms`: 时间戳（毫秒）
- `char_idx`: 对应字符索引（在 LyricLine.chars 中的位置）
- `checkpoint_idx`: 在该字符内的第几个节奏点（从 0 开始）

**相等性**：timestamp_ms、char_idx、checkpoint_idx 都相同

**约束**：
- 时间非负
- char_idx 在有效范围内
- 同一字符内的节奏点按时间递增

**与 TimeTag 的区别**：
- TimeTag 是时间标签的存储形式
- Checkpoint 是打轴时的操作单位
- 一个字符可以有多个 Checkpoint（连打场景）

#### CheckpointConfig（节奏点配置）

**职责**：配置每个字符的节奏点数量和类型

**属性**：
- `char_idx`: 字符索引
- `check_count`: 节奏点数量（需要击打几次，**默认 1，可以为 0**）
- `is_line_end`: 是否是句尾字符（行末，默认 False）
- `is_rest`: 是否是休止符（默认 False）
- `linked_to_next`: 是否与下一个字符连词（默认 False）

**业务含义**：
- `check_count`: 该字符对应的读音需要击打几次
  - **check_count >= 1**: 该字符有独立的节奏点，需要单独击打
    - 例：「赤」(あか) → check_count = 2（あ・か）
    - 例：「い」→ check_count = 1
  - **check_count = 0**: 该字符**没有独立节奏点**
    - 该字符和前一个字符连唱
    - 上一个字符的节奏点延续到该字符
    - 例：「ー」(长音) → check_count = 0，和前面的字符连唱
    - 例：「っ」(促音) → check_count = 0（如果配置不计数）
- `linked_to_next`: 标记当前字符与下一个字符连词
  - F3 快捷键切换此标记
  - 连词时下一个字符的 check_count 自动设为 0
  - 取消连词时下一个字符的 check_count 恢复为 1
  - 与 check_count=0 的区别：linked_to_next 是显式的连词标记，独立于 check_count
- `is_line_end`: 标记行末字符，通常有长音或休止
- `is_rest`: 特殊标记，表示这是一个休止符而非歌词字符

**连打场景示例**：
```
字符：  赤         ー    い    花
check：  2         0     1     1
击打：  [10][12]  (连)  [15]  [18]
        ↑   ↑     ↑
        あ   か    和"か"连唱（长音）
说明：「ー」没有独立时间标签，和「か」连唱
```

**相等性**：所有属性都相同

#### TimeTag（时间标签）

**职责**：表示一个时间点，**允许时间倒退**（用户修改时可能出现）

**属性**：
- `timestamp_ms`: 绝对时间（毫秒）
- `tag_type`: 标签类型（`char_start`, `char_middle`, `line_end`, `rest`）
- `singer_id`: 所属演唱者 ID（必填，用于多声部和和声场景）
- `char_idx`: 对应字符索引（在 LyricLine.chars 中的位置）
- `checkpoint_idx`: 在该字符内的第几个节奏点（从 0 开始）

**相等性**：timestamp_ms、tag_type 和 singer_id 都相同

**约束**：
- 时间非负
- 精度为毫秒
- **不强制递增**：允许时间倒退，以支持用户修改场景

**时间倒退场景**：
- 用户修改前面的歌词时间，导致比后面的晚
- 和声场景：不同演唱者的时间标签可能交错
- 系统不阻止添加，而是标记为"异常"供用户后续调整

#### Ruby（注音）

**职责**：表示汉字的注音（振り仮名）

**属性**：
- `text`: 注音文本（如 "あか"）
- `start_idx`: 起始字符索引
- `end_idx`: 结束字符索引（不包含）

**相等性**：所有属性都相同

**约束**：
- start_idx < end_idx
- 索引在有效范围内

## 业务规则详解

### 时间标签规则

**递增性（宽松规则）**：
- **不强制递增**：允许时间倒退，系统仅标记为"时间异常"供用户注意
- 同一行内的时间标签**建议**按时间递增，但不强制
- 行间时间标签可以重叠（多段ワイプ场景）
- 多演唱者场景：不同演唱者的时间标签完全独立，可任意交错

**对应关系**：
- 基础情况：一个字符对应一个时间标签
- 连打情况：一个字符对应多个时间标签（连续击打）
- 行尾休止：独立的时间标签表示行间停顿
- 多演唱者：每个演唱者有独立的时间标签集合

**精度**：
- 存储精度：毫秒（int）
- 显示精度：10 毫秒（0.01 秒）

**时间异常处理**：
- **检测**：系统检测时间倒退情况（后一个时间早于前一个）
- **标记**：在 UI 中高亮显示异常段落（如红色边框、警告图标）
- **渲染**：异常段落播放完成后高亮提示用户调整
- **不阻止**：不阻止用户继续打轴，允许先完成再调整

### 字符拆分规则

**日文文本**：
- 汉字：单独拆分（如「赤」）
- 假名：单独拆分（如「い」）
- 长音：可配置是否独立拆分（「ー」）
- 促音：可配置是否独立拆分（「っ」）

**英文文本**：
- 字母单独拆分
- 单词边界保留

**符号处理**：
- 标点符号单独拆分
- 连续空白合并为单个空格

### 注音规则

**覆盖规则**：
- 一个字符只能有一个注音
- 注音可以覆盖 1-N 个连续字符
- 支持多对一（如「昨日」→「きのう」）

**优先级**：
1. 用户手动指定的注音
2. 自动分析结果
3. 无注音（空）

### 节奏点（Checkpoint）规则

**数量计算（自动检查）**：

根据注音计算每个字符的节奏点数量：
- 汉字：注音的假名数量 = 节奏点数量
  - 例：「赤」(あか) → 2 个节奏点
  - 例：「花」(はな) → 2 个节奏点
- 平假名/片假名：通常 1 个
- 促音「っ/ッ」：可配置 0 或 1 个
- 长音「ー」：可配置 0 或 1 个
- 英文：可配置 1 个或多个

**句尾标记（is_line_end）**：
- 每行最后一个字符自动标记为句尾
- 句尾字符在导出时可能有特殊处理
- 支持手动调整句尾位置

**节奏点与时间标签的对应**：
- 基础情况：1 个节奏点 = 1 个时间标签
- 连打情况：1 个字符的 N 个节奏点 = N 个时间标签
- 未打轴：0 个时间标签（等待用户打轴）

**行尾休止**：
- 行末可配置休止符（独立的节奏点）
- 休止符在歌词显示中通常不显示
- 休止符用于控制行间停顿时间

### 字符与时间标签映射

**说明**：以下示例使用 `[分:秒.毫秒]` 格式表示时间，实际存储为毫秒整数（如 10000 毫秒 = 10.00 秒）

**一对一**：
```
字符：  赤        い        花
节奏点：1         1         1
时间：  [00:10.00] [00:15.00] [00:20.00]
        ↑10秒     ↑15秒     ↑20秒
```
- 「赤」在 10.00 秒开始唱
- 「い」在 15.00 秒开始唱
- 「花」在 20.00 秒开始唱

**一对多（连打）**：
```
字符：  赤              い        花
节奏点：2               1         1
时间：  [00:10.00]     [00:15.00] [00:20.00]
        [00:12.00]
        ↑   ↑
        あ   か（2个节奏点：10.00秒和12.00秒）
```
- 「赤」有两个节奏点：10.00秒（あ）和 12.00秒（か）
- 「い」和「花」各有一个节奏点

**多对一（连音）**：
```
字符：  昨        日
节奏点：2（きのう）  0（连读）
注音：  きの+      う
时间：  [00:10.00] [00:15.00]
        ↑
        きのう跨越两个字
```
- 「昨」和「日」对应注音「きのう」
- 「きの」打在 10.00 秒（对应「昨」）
- 「う」打在 15.00 秒（对应「日」）
- 注意：实际唱的时候是一个整体

**连打场景（check_count = 0）**：
```
字符：  赤         ー        い        花
节奏点：2          0         1         1
时间：  [00:10.00]          [00:15.00] [00:20.00]
        [00:12.00]
        ↑   ↑     ↑连读
        あ   か    「ー」和「か」连唱
```
- 「赤」有两个节奏点：10.00秒（あ）、12.00秒（か）
- 「ー」check_count = 0，没有独立时间，和「か」连唱
- 「い」在 15.00 秒
- 「花」在 20.00 秒

**存储格式**：
```python
# 实际存储为毫秒整数
赤.timetags = [
    TimeTag(timestamp_ms=10000),  # 10.00 秒
    TimeTag(timestamp_ms=12000)   # 12.00 秒
]
い.timetags = [
    TimeTag(timestamp_ms=15000)     # 15.00 秒
]
```

### 多演唱者（和声）规则

**演唱者划分**：
- 默认只有一个演唱者（演唱者1）
- 用户可添加多个演唱者（演唱者2、3...或命名如"合唱"、"和声"）
- 每个演唱者有独立的：
  - 歌词行集合
  - 时间标签集合
  - 显示颜色

**同时演唱场景**：
```
时间点：[10秒]
演唱者1：赤（高亮红色）
演唱者2：青（高亮蓝色） ← 同时高亮

时间点：[15秒]
演唱者1：い（高亮红色）
演唱者2：（无歌词）
```

**时间重叠处理**：
- 不同演唱者的时间标签完全独立
- 允许同一时间点多个演唱者同时有高亮字符
- UI 使用不同颜色区分不同演唱者
- 时间倒退仅在同演唱者内检测，跨演唱者不检测

**演唱者切换**：
- 用户可选择当前为哪个演唱者打轴
- 支持"全部显示"模式（查看所有演唱者）
- 支持"单演唱者"模式（专注一个轨道）

## 聚合边界

### Project 聚合

**聚合根**：Project
**包含实体**：LyricLine, Singer
**包含值对象**：TimeTag, Ruby, Checkpoint, CheckpointConfig

**一致性规则**：
- 所有修改必须通过 Project 进行
- 删除 Project 时级联删除所有行和演唱者
- 删除演唱者时：级联删除其所有歌词行，或转移给其他演唱者
- 行的顺序由 Project 维护（可按演唱者分组）
- 必须至少保留一个演唱者

## 验证规则

### 项目级验证

1. **音频文件存在性**：audio_path 指向的文件必须存在
2. **歌词行非空**：至少有一行歌词
3. **时间标签完整性**：每行至少有一个时间标签（除空行外）

### 行级验证

1. **时间递增性**：时间标签时间戳递增
2. **数量合理性**：时间标签数量不超过字符数量过多
3. **注音覆盖性**：注音索引不越界

### 时间标签验证

1. **非负性**：时间戳 >= 0
2. **合理性**：时间戳不超过音频时长

## 音频时长验证

### 用途

虽然项目文件不存储音频路径，但存储音频时长（`audio_duration_ms`）用于：
- 提示用户选择的音频是否与原项目匹配
- 时长差异过大时显示警告（但不阻止使用）

### 验证逻辑

```python
def validate_audio_match(project: Project, audio_path: str) -> Tuple[bool, str]:
    """
    验证用户选择的音频是否与原项目匹配
    返回: (是否通过, 提示信息)
    """
    audio_duration = get_audio_duration(audio_path)
    stored_duration = project.audio_duration_ms
    
    if stored_duration == 0:
        return True, "新项目，无音频时长记录"
    
    diff_ratio = abs(audio_duration - stored_duration) / stored_duration
    
    if diff_ratio < 0.01:  # 差异 < 1%
        return True, "音频匹配"
    elif diff_ratio < 0.05:  # 差异 < 5%
        return True, "音频时长略有差异，可能为不同版本"
    else:
        return False, "音频时长差异过大，请确认是否为正确的音频文件"
```

### 用户交互

- 差异 < 1%：静默通过
- 差异 1-5%：显示提示但不阻止（"音频可能为不同版本，是否继续？"）
- 差异 > 5%：显示警告，用户确认后仍可继续

## 非日语歌词处理

### 语言检测

- 通过 `metadata.language` 字段标识
- 支持语言：`ja` (日语), `zh` (中文), `en` (英文), 等

### 注音支持

- **日语**：完整注音支持（RubyAnalyzer）
- **中文**：可选拼音注音（扩展点）
- **英文**：单词级注音（可选，显示音标）
- **其他**：禁用注音功能

### 字符拆分

- **日语**：汉字、假名、长音、促音分别处理
- **中文**：按字拆分（每个汉字一个字符）
- **英文**：按字母拆分，支持连字符（如 don't → do + n't）
